Архитектура проекта

Проект построен с использованием многослойной архитектуры и разделён на следующие логические слои:

Domain
Содержит доменную модель предметной области.
Включает интерфейсы IAlive и IInventory, иерархию животных (Animal, Herbo, Predator и конкретные виды), а также иерархию инвентаризируемых вещей.
Данный слой не зависит от других слоёв системы.

App
Содержит бизнес-логику и сценарии использования системы.
Включает сервис ZooService, который реализует основные операции зоопарка, а также интерфейсы IVeterinaryClinic и IAnimalRepository.
Работает исключительно через абстракции и зависит только от Domain слоя.

Infrastructure
Содержит конкретные реализации интерфейсов Application слоя.
Включает реализацию ветеринарной клиники, in-memory репозиторий животных и конфигурацию DI-контейнера.
Позволяет легко заменить технические реализации без изменения бизнес-логики.

ConsoleApp
Содержит пользовательский интерфейс приложения.
Реализует консольное меню, ввод и вывод данных, а также взаимодействие с Application слоем.
Не содержит бизнес-логики.

Tests
Содержит юнит-тесты, проверяющие корректность бизнес-логики и доменных правил.
Используются фейковые реализации интерфейсов для изоляции тестируемых компонентов.

Применение принципов SOLID

Single Responsibility Principle
Каждый класс имеет одну зону ответственности.
Доменные классы описывают состояние сущностей, сервисы реализуют бизнес-логику, ConsoleApp отвечает только за ввод и вывод данных.

Open/Closed Principle
Система открыта для расширения и закрыта для модификации.
Добавление новых видов животных или реализаций сервисов не требует изменения существующего кода.

Liskov Substitution Principle
Все наследники базового класса Animal могут использоваться вместо него без нарушения корректности работы программы.

Interface Segregation Principle
Интерфейсы минимальны и специализированы.
IAlive отвечает только за потребление еды, IInventory — только за инвентаризацию.

Dependency Inversion Principle
Высокоуровневые модули не зависят от низкоуровневых реализаций.
ZooService зависит от абстракций IVeterinaryClinic и IAnimalRepository, которые внедряются через DI-контейнер.

Dependency Injection

В проекте используется встроенный DI-контейнер Microsoft.Extensions.DependencyInjection.
Регистрация всех зависимостей вынесена в отдельный класс конфигурации в Infrastructure слое.
Это упрощает тестирование и повышает расширяемость системы.

Юнит-тестирование

Для тестирования используется фреймворк xUnit.
Тестируется бизнес-логика сервисов и доменные правила.
Используются фейковые реализации интерфейсов.
Обеспечено покрытие ключевых сценариев работы приложения.
